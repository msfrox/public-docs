# Sakina — Free Sri Lankan Muslim Matrimony

A **free**, private, halal matrimony service for the Sri Lankan Muslim community.
No fees, no photos stored, contact shared only when both sides agree.

🌐 **Live:** [sakina.lk](https://sakina.lk) · **Status:** deployed and verified end-to-end
(real OTP signup works). Not yet publicly promoted — Tamil/Sinhala await native review and
a final palette is being chosen.

> This README is the project handover document — architecture, how it works, how to
> run/deploy it, and the version history. **Pending work lives in [`BACKLOG.md`](./BACKLOG.md);**
> deployment specifics are in [`DEPLOY.md`](./DEPLOY.md).

## What it is

A request-based, identity-verified matrimony platform. The flow:

> Sign up with a Sri Lankan mobile (Turnstile + OTP) → build a profile → an admin
> approves it → browse matches → express interest → the other person sees your full
> profile **except your number** → if they accept, **both** receive each other's
> WhatsApp (link + QR) → continue on WhatsApp.

## Design principles

1. **Free, forever.** No paywalls, no "unlock contact" fees.
2. **Privacy first.** Phone numbers are encrypted at rest and never appear in browse
   responses; contact is revealed only on mutual accept (and every reveal is audited);
   no photos are stored; profiles are `noindex` (never on Google); conversations live
   on WhatsApp, not our servers.
3. **Identity instead of money.** A verified Sri Lankan SIM (NIC-registered by law) +
   OTP gates abuse, one account per number — replacing the usual paywall filter.
4. **Mostly hands-off.** A light manual profile-approval step, then an automated
   trust/abuse engine (silent rate limits, auto-hide/suspend) + reporting + appeals.
5. **Halal-aligned.** No open chat (request → accept), modesty by default, wali-friendly
   (women are nudged to use a guardian's number).
6. **Mobile-first + installable (PWA).**
7. **PDPA-compliant** (Sri Lanka Personal Data Protection Act No. 9 of 2022): consent,
   data export, erasure, and appeal of automated decisions.
8. **Multilingual** — the client UI is available in English, Tamil, and Sinhala.

## Tech stack

| Layer | Choice |
|-------|--------|
| Frontend | React 19 + Vite + Tailwind v4, as a mobile-first **PWA** (static assets) |
| API | **Cloudflare Workers + Hono** (`/api/*`) |
| Database | **Neon Postgres** via **Cloudflare Hyperdrive** (Drizzle ORM) |
| Auth | Phone + OTP (**Text.lk** SMS), server-side sessions (httpOnly cookie) |
| Bot protection | **Cloudflare Turnstile** on OTP |
| Languages | English + **Tamil** + **Sinhala** (lightweight custom i18n, code-split locales) — see [Internationalization](#internationalization) |
| Theming | Swappable palette tokens (`--p-*`, 4 palettes + dark-mode groundwork); self-hosted Playfair Display + Poppins — see [Theming & branding](#theming--branding) |
| Hosting | One Cloudflare Worker serves the PWA + API; custom domain `sakina.lk` |
| Source | Private repo `github.com/msfrox/Sakina` (worker name stays `nikah`) |

Everything runs on Cloudflare's free tier (Phase 1). One Worker serves the React app
as static assets and handles `/api/*`, so page views don't consume the request budget.

## How protection works (security model)

- **App-layer authorization** on every endpoint (the primary control): each request is
  permission-checked server-side and returns only allowed fields (per-endpoint DTOs).
- **Parameterised queries** (Drizzle) — no SQL injection.
- **Contact numbers**: stored AES-GCM encrypted; the account number is also keyed-HMAC
  hashed for lookup. Decrypted only at the reveal step for an accepted pair; never in
  any list/browse payload. UUID primary keys (no enumeration).
- **Sessions**: random token, only its SHA-256 stored; httpOnly + Secure (prod) + SameSite.
- **Trust/abuse engine** (`worker/lib/trust.ts`): scores accounts and auto-throttles,
  hides, or suspends; suspended users can appeal (PDPA).
- **No contact in free-text:** a profile whose free-text fields (about / family / custom)
  contain a phone-number-like string is **rejected on submit** (server + client), so a
  number can't leak outside the mutual-accept reveal flow.
- **Production secrets** live only in Cloudflare (write-only) **and** a password manager;
  they're never committed and never generated by an AI agent — see
  [Secrets & disaster recovery](#secrets--disaster-recovery).
- **Deferred hardening:** Postgres Row-Level Security (a database-level backup layer
  against future app bugs) — planned as a post-launch pass (tracked in `BACKLOG.md`).

## Profile fields

- **Essential (always shown):** name, gender, date of birth, marital status, children,
  district, willing-to-relocate, contact (WhatsApp).
- **Built-in (admin-editable — relabel / reorder / require / hide):** education,
  profession, work location, income, height, languages, family background, about,
  looking-for. Managed in **Admin → Field manager → Built-in fields**.
- **Custom (admin-created):** any extra field (e.g. ethnicity, religious practice),
  stored in `profiles.attributes` (JSONB).
- **Reference number:** every profile gets a human-friendly id **`YYMM-NNN`** (e.g.
  `2606-001`) at creation; the sequence resets each month (via the `seq_counters` table),
  so the running total is never exposed. Shown on the admin cards/detail.

> *Note:* a "complexion/skin-tone" field is deliberately omitted to avoid colorism.

## Maintaining the site (admin operations)

Sign in with an admin number, then open **/admin**. Tabs: **Overview** (analytics),
**Approval queue**, **All profiles** (every profile, any status), **Appeals**,
**Reports**, **Field manager**. The queue and All-profiles list each have a **View
profile** button → a full-detail modal so you can vet a member's content.

**Add / remove an admin.** Admins are config-driven, not clicked in the UI (safer).
Edit the `ADMIN_PHONES` secret (comma-separated E.164 numbers) and that person becomes
an admin the next time they log in:
```
npx wrangler secret put ADMIN_PHONES
# paste e.g.  +94755655166,+94770000000
```
Removing a number revokes admin on their next request.

**Ban / unban a member.** Each profile (in the queue, the All-profiles list, and the
detail modal) has a **Ban** button. Banning **prompts for a reason** (kept on the
account + audit log; shown to admins — not yet shown to the member, see `BACKLOG.md`),
sets the account to banned, hides the profile, blacklists the number (can't re-register),
and revokes their sessions. Unban reactivates the account, clears the reason, and
re-queues the profile for review.

**Analytics.** The **Overview** tab shows live counts (members, new signups 7/30d, live
profiles, matches, reports, suspended/banned) — pulled from our own DB, no third party.

**Control what each field shows.** **Field manager → Built-in fields**: for every field
(including **name**) set its visibility — **In search** (browse cards), **Profile page**
(only when a profile is opened), or **After match** (only once two people match). You can
also relabel, reorder, require, or hide built-in fields, and create custom fields with
their own visibility. `name` defaults to *Profile page* (not shown in search results).

**SMS sender ID.** Approved Text.lk sender IDs: **`SakinaOTP`** (OTP), plus **`Sakina`**
and **`Sakina LK`** (reserved for future non-OTP messages). To set/change it:
```
npx wrangler secret put TEXTLK_SENDER_ID    # paste: SakinaOTP
```

**Rotate a secret** (e.g. Turnstile or Text.lk token): `npx wrangler secret put <NAME>`.

**Ship a change:** bump `SITE_VERSION`, add a Version-history row, then `npm run deploy`.

> ⚠️ **If the change includes a new migration, run `npm run db:migrate:prod` BEFORE
> `npm run deploy`.** Deploying first means the new worker queries columns the DB doesn't
> have yet → 500s on the affected endpoints. (This bit us once with migration `0004`.)

**Database capacity (Neon free tier).** 0.5 GB storage ≈ tens of thousands of text-only
profiles; 100 compute-hours/month (the DB auto-sleeps when idle) and 5 GB transfer easily
cover ~1,000 new users/month. Watch the **Compute** meter in the Neon dashboard; upgrade
the plan only when growth/uptime needs it — storage won't be the first limit.

## Internationalization

The whole **client-facing** UI is available in **English, Tamil, and Sinhala**. Admin is
intentionally English-only.

- **Lightweight, no library** (`src/i18n/`). English (`en.ts`) is bundled and is the
  source of truth; **Tamil (`ta.ts`) and Sinhala (`si.ts`) are code-split** and lazy-loaded
  only when selected (~6 KB gz each), so English users pay ~0 extra.
- A **language toggle** sits in the top bar of every client page. The choice is persisted
  (`localStorage: sakina.lang`), `<html lang>` is kept in sync, and any missing key falls
  back to English.
- **Edit copy:** change `src/i18n/en.ts` (source), then mirror the same key in `ta.ts` /
  `si.ts`. Use `t('section.key')` in components (`const t = useT()`).
- **Add a language:** add a `xx.ts` locale, a loader in `i18n/index.tsx`, and an entry in
  `LANGS`.
- ⚠️ The Tamil/Sinhala strings are **AI-drafted and pending native-speaker review** before
  the site is shared publicly (flagged in the `ta.ts` / `si.ts` headers).

## Theming & branding

The look is **festive / wedding-invitation** themed and fully **re-skinnable**.

- **Swappable color tokens.** All colors flow through CSS variables `--p-*` defined in
  `src/index.css`; Tailwind's `brand-*` / `accent-*` / `sand-*` / `ink` / `muted` map to
  them (`@theme inline`). Changing the palette recolors the entire app with no per-page edits.
- **4 palettes** via `data-palette` on `<html>`: **teal** (default), **plum**, **maroon**,
  **emerald**. A floating **Theme** switcher (`src/components/PaletteSwitcher.tsx`) lets you
  compare them live — it renders **only on localhost** (dev), never for real visitors.
- **Lock a final palette:** copy that palette's `--p-*` values into `:root` in `index.css`,
  then **delete `PaletteSwitcher`** (remove it from `src/components/Root.tsx`).
- **Fonts** (self-hosted woff2, preloaded, `swap`): **Playfair Display 600** for headings
  (the invitation feel) + **Poppins 600** for the logo/UI; body uses the sans stack. (Load
  a 2nd Playfair weight if faux-bold on `font-bold` headings looks off.)
- **Ornaments:** inline-SVG motifs in `src/components/Ornaments.tsx` (Mughal arch, 8-point
  star, divider, star-behind-leaf sprig), recolored via `currentColor`.
- **Dark mode:** groundwork done (every color is a `--p-*` var). To ship: add a
  `[data-theme='dark']` block overriding the neutral vars + a toggle.

## Secrets & disaster recovery

- **Dev:** gitignored `.dev.vars` (copy from `.dev.vars.example`). **Prod:** set with
  `wrangler secret put` — they live only in Cloudflare, which is **write-only** (you cannot
  read a secret back).
- **RULE — never let an AI agent generate or see PRODUCTION secret values.** Generate them
  yourself locally with a CSPRNG and set them yourself (also recorded in the global
  `~/.claude/CLAUDE.md`):
  ```
  [Convert]::ToBase64String([System.Security.Cryptography.RandomNumberGenerator]::GetBytes(32))
  ```
- ⚠️ **Back up prod secret *values* in a password manager** — especially **`PHONE_ENC_KEY`**
  (decrypts every stored phone number) and **`PHONE_HASH_KEY`**. No readable backup = if the
  Cloudflare secret is ever lost, the encrypted data is unrecoverable. (Prod keys were
  rotated to known, backed-up values on 2026-06-09.)
- The account number is stored **both** keyed-HMAC hashed (lookup) **and** AES-encrypted
  (`users.phone_enc`). `PHONE_HASH_KEY` effectively **can't be rotated once real users
  exist** without decrypting + re-hashing every row inside the app.
- `TEXTLK_API_TOKEN` and the Turnstile secret are recoverable from the Text.lk / Cloudflare
  dashboards if lost.

## Repository & security

- **Private repo** `github.com/msfrox/Sakina` (branch `master`); GitHub account has **2FA**.
- `.gitignore` excludes `dist`, `.env*`, `.dev.vars*`, `.wrangler` — **no secrets or build
  artifacts are tracked**.
- A versioned **pre-commit secret guard** (`.githooks/pre-commit`) blocks staged
  `.env`/`.dev.vars` files and high-signal secret patterns. It auto-enables on `npm install`
  via the `prepare` script (`git config core.hooksPath .githooks`).
- **Dependabot** alerts + automated security fixes are on. (GitHub secret scanning/push
  protection isn't free for private repos, so the pre-commit hook is the stand-in.)
- **Deploy from GitHub (CI/CD)** via a `cloudflare/wrangler-action` workflow is planned —
  see `BACKLOG.md`.
